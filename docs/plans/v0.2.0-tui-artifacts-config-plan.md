# Praxis v0.2.0 TUI, Artifact, Config, and Release Readiness Implementation Plan

> **For Hermes:** Use subagent-driven-development skill to implement this plan task-by-task.

**Goal:** Prepare a credible v0.2.0 server-accessible Praxis build by fixing the real-world eval defects, wiring TUI chat to the backend, adding TUI project/backlog/config/artifact workflows, and hardening state-grounded artifact generation.

**Architecture:** Consolidate CLI and TUI backend setup into reusable service modules, then make both interfaces call those services rather than duplicating logic. Keep Praxis local-first: configuration lives in `PRAXIS_HOME`, engagement data lives under `<project>/.praxis`, artifacts are files in `<project>/.praxis/artifacts`, and every mutation goes through existing repo/config APIs with audit events.

**Tech Stack:** Python 3.11+, Typer, Textual, Rich, Pydantic v2, SQLite, YAML/Markdown, pytest/Textual Pilot/respx. Tests must not hit real APIs.

---

## Ground rules

1. Follow TDD for every code change: write a failing test first, run it, implement, run it green.
2. Do not use real API keys in tests. Use fake transports or monkeypatch `make_transport`.
3. Do not bypass repositories/config loaders in TUI. TUI mutations must call the same backend services used by CLI.
4. Keep all generated user artifacts as files under `<engagement>/.praxis/artifacts/...` and return both rendered content and absolute file path.
5. Keep this branch server-accessible at `/root/praxis-realworld-eval/repo` and install/test using `uv run --project /root/praxis-realworld-eval/repo praxis ...`.

2. Phase 2 shared chat runtime.
